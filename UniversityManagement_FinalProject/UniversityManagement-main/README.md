# University Management System (SMS) — RESTful API Сервіс

Цей проєкт є повнофункціональним архітектурним рішенням для цифрової екосистеми вищого навчального закладу. Застосунок є сучасним **RESTful API**, побудованим на базі фреймворку Spring Boot. В основі розробки лежать принципи багатошарової структури, суворе дотримання REST-конвенцій, використання незмінних об'єктів для передачі даних та автоматизоване тестування.

## 🧱 Компоненти технологічного стека
* **Платформа розробки:** Java (версія 21 або новіша)
* **Ядро системи:** Spring Boot 3.x (модулі Spring Web, Spring Data JPA, Spring Boot Validation)
* **Простір збереження даних:** PostgreSQL (основна виробнича СУБД) та H2 Database (швидка embedded-база для ізольованого виконання тестів)
* **Автоматизація та збірка:** Gradle
* **Інтерактивна документація:** Springdoc-OpenAPI v2 (інтерфейс Swagger UI)
* **Контроль якості:** JUnit 5, інструменти MockMvc та AssertJ

## 🏗️ Архітектурна організація проєкту
У додатку реалізовано класичну **багатошарову архітектуру (Layered Architecture)**, яка гарантує ізоляцію бізнес-логіки від зовнішніх інтерфейсів та шару доступу до бази даних:

* `src/main/java/ua/university/sms/controller/`   --> Шар REST-контролерів: обробка вхідних HTTP-запитів та перевірка DTO
* `src/main/java/ua/university/sms/service/`      --> Шар бізнес-логіки: обчислення рейтингів, реєстраційні процеси, фільтри
* `src/main/java/ua/university/sms/repository/`   --> Шар доступу до даних: робота з базою через інтерфейси Spring Data JPA
* `src/main/java/ua/university/sms/model/entity/`  --> JPA-сутності (Student, Teacher, Course, Enrollment)
* `src/main/java/ua/university/sms/model/dto/`     --> Об'єкти переносу даних (сучасні незмінні Java Records для запитів/відповідей)
* `src/main/java/ua/university/sms/exception/`    --> Шар глобального перехоплення та обробки винятків (@ControllerAdvice)

## ⚙️ Специфікація REST API Ендпоінтів

Сервіс надає повний спектр операцій маніпулювання даними (CRUD) для чотирьох основних бізнес-доменів:

### 1. Адміністрування студентів (/api/students)
* `POST /api/students` — реєстрація нового студента із перевіркою полів (стартовий GPA автоматично дорівнює 0.0).
* `GET /api/students` — вибірка списку студентів із можливістю динамічної Query-фільтрації за допомогою параметрів:
  * За поточним статусом: ?status=Active
  * За курсом навчання: ?year=2
* `GET /api/students/{id}` — перегляд повної картки студента за ідентифікатором.
* `PUT /api/students/{id}` — повне оновлення персональних даних за допомогою об'єкта StudentRequest.
* `DELETE /api/students/{id}` — безпечне видалення студентського профілю із бази.
* `GET /api/students/search` — регістронезалежний (Case-Insensitive) пошук студентів за частиною ПІБ або адресою електронної пошти за допомогою параметра ?query=...
* `GET /api/students/top` — формування регістру топ-10 студентів за найвищим показником середнього бала (GPA).

### 2. Керування викладацьким складом (/api/teachers)
* Надає вичерпний набір базових CRUD-методів (GET, POST, PUT, DELETE) для ведення обліку професорів та викладачів кафедр.

### 3. Менеджмент навчальних курсів (/api/courses)
* Повний набір CRUD-операцій для обліку дисциплін із гнучким закріпленням за відповідальним викладачем.
* `GET /api/courses` — фільтрація переліку доступних дисциплін за ідентифікатором викладача (?teacherId=...) або за обсягом кредитів ECTS (?credits=...).

### 4. Навчальний процес: Зарахування та Успішність (/api/enrollments)
* `POST /api/enrollments` — фіксація запису студента на обраний курс (параметр фінансового статусу paid за замовчуванням отримує значення false).
* `PUT /api/enrollments/{id}/grade` — виставлення оцінки за дисципліну за допомогою параметра ?grade=... При цій операції система автоматично запускає тригер повного перерахунку та оновлення підсумкового середнього бала (gpa) у профілі студента.
* `PUT /api/enrollments/{id}/payment` — зміна та фіксація статусу оплати за навчання на курсі за допомогою параметра ?paid=...
* `GET /api/enrollments/unpaid-students` — отримання списку студентів, які мають борг з оплати хоча б за один обраний курс.
* `GET /api/enrollments/course-average/{courseId}` — розрахунок та виведення поточної середньої успішності всіх студентів у межах конкретної дисципліни.

## 🛡️ Надійність, Валідація та Обробка помилок

1. **Строгий контроль вхідних даних:** Валідація інформації реалізована безпосередньо на вході в контролери за допомогою специфікації `jakarta.validation`. Використовуються перевірки форматів email (регулярні вирази), обмеження на порожні поля (@NotBlank), контроль розмірів текстових блоків та лімітів для кредитного навантаження.
2. **Централізований менеджмент винятків:** У застосунку відсутня стандартна сторінка помилок сервера (White Label Error Page). Завдяки компоненту `GlobalExceptionHandler` з анотацією `@ControllerAdvice`, будь-який збій трансформується у зрозумілий клієнту JSON-об'єкт із деталізацією причини та валідним HTTP-статусом:
   * `400 Bad Request` — помилки валідації полів або некоректний формат вхідного тіла запиту;
   * `404 Not Found` — відсутність запитуваного ресурсу в базі даних (генерується через кастомний ResourceNotFoundException);
   * `500 Internal Server Error` — непередбачувані технічні або системні аномалії на боці сервера.

## 🛠️ Порядок локального розгортання

### Крок 1. Підготовка сховища даних
Застосунок використовує промислову СУБД **PostgreSQL**. Перед запуском переконайтеся, що на вашому локальному сервері створено порожню схему з назвою `university_db`, а конфігураційні параметри у файлі `src/main/resources/application.properties` відповідають вашим доступам:

* spring.datasource.url=jdbc:postgresql://localhost:5432/university_db
* spring.datasource.username=ваш_користувач_бази
* spring.datasource.password=ваш_пароль_бази
* spring.jpa.hibernate.ddl-auto=update

### Крок 2. Компіляція та запуск додатка
Відкрийте консоль у кореневому каталозі проєкту та виконайте команду для автоматичної збірки й запуску вбудованого сервера:

```bash
./gradlew bootRun
```

Після успішного старту веб-сервер Tomcat почне приймати вхідні запити за адресою: http://localhost:8080.

### Крок 3. Інтерактивна документація (Swagger UI)

Завдяки інтегрованій бібліотеці Springdoc-OpenAPI, ви можете тестувати всі ендпоінти безпосередньо через браузер. Повна специфікація з прикладами JSON-структур доступна за посиланням: http://localhost:8080/swagger-ui.html

## 🧪 Автоматизоване тестування

Якість коду та логіка математичних розрахунків (включаючи калькуляцію GPA) покриті пакетом із 10 комплексних інтеграційних тестів на базі `MockMvc`.

Тести повністю ізольовані від вашого робочого середовища: вони використовують незалежний файл конфігурації у папці тестів (`src/test/resources/application.properties`) та виконуються у пам'яті за допомогою H2 Database (In-Memory), що гарантує збереження локальної PostgreSQL від появи тестового сміття.

Для запуска тестових сценаріїв та отримання візуального звіту виконайте команду:

```bash
./gradlew clean test
```
